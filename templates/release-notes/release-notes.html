Codebase

Soft Delete for Applications in Workspaces

  • The ability for users to delete workspaces has been a feature on our website for quite a while but the way this was being handled on the backen of our system is not necessarily the best way to have architecturally done so. Many pages that were related to that deleted application would still be accessibl, which is not very secure. In this release we did some work to implement archiving applications in workspaces. When an application is deleted now, the application as well as many other pages such as the run and information pages are no longer accessible. In the case a user deletes an application accidentally or realizes they still need that application, they can reach out to us via any of the contact method listed on our contact us page and we can re-instate that very application in their workspace with the same run history if needed.

Out of Service Workspace Applications Removed

  • While we were in the past still working on how applications are referenced when new infrastructure is created this cause applications that some users added to their workspaces to become unresponsive. We ave systematically removed all applications from user workspaces that were in an unusable state. Users might have found that these applications were making the site run more slowly for them as well as were unable to remove them from their workspaces. All this should now be fixed.

Safer Error Logging

  • In the past when our infrastructure and code would detect errors that we did not account our code for, the error would be logged to the browser potentially revealing sensitive information thats should not be available to users. We've implemented a change that will catch these unknown errors and anonymize them after the original log is saved to our servers and send the new error message to the browser. We've made this change in order to both keep our users safe as well as to more easily investigate errors that users may run into or report to us.

New Workflows

Bidding to Build a Part

During this release we worked to try to transform how manufacturers find the right supplier for a part they wish to build. Larger manufacturers may have a few hundred to a couple thousand suppliers, which makes it hard to find the right supplier that can machine a newly designed part. We envision the DMC can be used in the following way to increase productivity:
  1. Manufacturer uploads a .stp file to their private workspace documents page.
  2. The manufacturer then uses the Technical Data Packaging Tool application in their private workspace choosing to use the file they've uploaded as an input to the application.
    • They also specify the coating and material type at this stage.
  3. The manufacturer then creates a new private workspace and invites all their suppliers to it.
    • They will also share authorization tokens with their suppliers, which are used to run the Manufacturing Should Cost Analysis application.
    • The tokens are also used so that suppliers are not able to gain the system.
  4. The manufacturer then shares the output of the TDP application to their new workspace and allows the files to be accessible to team members of the new workspace.
  5. Newly invited suppliers on the DMC have their organization administrators fill out their organization VPCs.
  6. Individual suppliers now created their own private workspaces and add the Manufacturing Should Cost Analysis application to it.
  7. Suppliers now run the Manufacturing Should Cost Analysis application and use the Manufacturer's supplied file as an input from that shared workspace.
    • Suppliers at this time will use their one time use authorization tokens to run the application.
    • This will generate a report that can then be saved to their workspace documents page and can then be reviewed before sharing. Suppliers will be give three authorization tokens that allow them to update their VPC in the case they realize they may have entered some information in incorrectly.
  8. Suppliers now share the Should Cost Report from their workspace to a new workspace the Manufacturer has created that is only accessible to the Manufacturer. This means the reports are not viewable by any other competitor.
  9. The manufacturer now can make an informed decision on which suppliers have the actual capabilities to create the part they are looking to have made and also know which suppliers may be to costly to use in this case. They can now figure out which suppliers they may want talk to before making their final decision.

Codebase

Vault Integration

  • We've integrated Vault into our deployment framework for our playform in this release. This change will not effect users directly but makes sure that our team is able to work on our platform virtual machines in a much safer capacity. Users as usual can rest assured that only those with the proper access level are working on updating the code base and infrastructure that supports the website.

Updated Terraform Infrastructure Deployment Scripts

  • Users that know our development team's infrastructure deployment methods know that we use Terraform to deploy and build our infrastructure. We've made some updates due to some recent changes that allows us to once more deploy our infrastructure in a more automated way.
  • Along with this update, we now have new code that allows us to now automatically build our ELK Stack/Stack Monitoring Infrastructure in tandem with the rest our our infrastructure.

Updated Terraform Version

  • We've updated our Terraform deployment scripts to now be compliant with Terraform Verson 0.9.4. If you are interested in deployiong our infrastructure yourself for this or other versions of the DMC going forward, you will need to make sure you have this version of Terraform installed.

File Signing (SHA256)

  • In order to give users the piece of mind that they are not downloading documents that have been altered after they have been uploaded to our site, we have implemented SHA256 for uploaded files. SHA256 is a cryptographic hash which is like a semi unique signature for a text or a data file. The signature is very difficult to regenerate exactly the same way once created, so our users can know that the files being downloaded are always the original and unaltered versions uploaded to the DMDII Projects' profile pages.

Quarantine Bucket Workflow

  • In a previous release we implemented a framework for our quarantine bucket. We've now finalized and fully tested its functionality and have included it in this release. In the past, our infrastructure would scan all documents being uploaded to our site before making the document available for user to download. Documents that did not pass the virus and malware scan process would be marked as not safe and would not become available for download through our site. This will move files marked as not safe by our infrastructure will be moved to our quarantine storage solution, when users go to retrieve the document the next time they will find in place of the document an image informaing them the file was not properly uploaded. We will at this point be able informed of the action so we can analyze the threats our site experiences on a regular bases, to ensure our users are always safe when using our platform.
